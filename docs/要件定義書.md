# QRコード受付システム 要件定義書

## 1. プロジェクト概要

### 1.1 背景・目的

| 項目 | 内容 |
|------|------|
| **対象イベント** | 留学生新入生オリエンテーション（生活ガイダンス） |
| **規模** | 毎学期 約150〜200名 |
| **課題** | 学生証未発行のため、紙・スプレッドシート名簿の Ctrl+F 検索で受付していた |
| **解決策** | QRコードによる受付システムで効率化 |
| **大前提** | QRコードメールを送付した参加者のみが来場する。QRコードは本学が振るIDに紐づき、そのIDで受付を行う |

### 1.2 最適化対象

オリエンテーション初日の受付運用に最適化する。

## 2. 技術スタック

### 2.1 バックエンド・メール送信

| 技術 | 用途 |
|------|------|
| **Google Apps Script (GAS)** | スプレッドシート連携、メール送信、Web API |
| **Google スプレッドシート** | 参加者マスターデータ、メール設定 |
| **Gmail** | 下書き作成・一斉送信 |
| **GoQR API** | QRコード画像生成（`https://api.qrserver.com/v1/create-qr-code/`、HTTPS 使用） |

### 2.2 フロントエンド（QR読み取りアプリ）

| 技術 | 用途 |
|------|------|
| **HTML5** | アプリ基盤 |
| **GitHub Pages** | ホスティング |
| **JavaScript** | カメラ制御、QR読み取り、UI制御 |
| **QRコード読み取りライブラリ** | html5-qrcode を推奨（CDN: unpkg.com。カメラAPI内蔵、カスタムUI対応）。詳細は [技術選定チェック結果](./技術選定チェック結果.md) 参照 |

### 2.3 データ連携

| 連携 | 方式 |
|------|------|
| フロントエンド ↔ 参加者データ | GAS Webアプリ（doGet / doPost）経由のREST API |

## 4. 機能要件

### 4.1 GAS（バックエンド）機能

| 機能 | 内容 |
|------|------|
| **スプレッドシート初期設定** | **メニューから実行すると、参加者データシート・メール設定シートを自動作成し、ヘッダー行・項目構成を設定する。手動でのシート作成は不要** |
| 参加者データ管理 | スプレッドシートでのCRUD |
| メール設定 | ダイアログで設定、シートに保存 |
| メール本文 | HTML形式、氏名・学籍番号・所属課程・IDを差し込み |
| QRコード挿入 | GoQR APIで参加者IDをエンコードしたQR画像を本文に挿入 |
| 下書き一括作成 | 参加者データの宛先に対してGmail下書きを一括作成 |
| 下書き作成日時記録 | 一括作成時に参加者データの「下書き作成日時」を更新 |
| 一斉送信 | 下書きを一括送信 |
| **送信完了時の更新** | **一斉送信完了時に、参加者データの「送信日時」「送信ステータス」を必ず更新する** |
| 参加者データ取得API | doGet：IDによる参加者情報取得（午前・午後それぞれの受付データを返す） |
| 受付登録API | doPost：スキャン時に午前または午後の受付ステータス・受付日時・受付担当者をスプレッドシートに書き戻す（modeパラメータで午前/午後を指定） |

### 4.2 メール送信フロー

1. 参加者データにメールアドレスを入力
2. メール設定ダイアログで本文・送信者等を設定
3. 「下書き一括作成」でGmailに下書きを作成
4. 下書き確認後、「一斉送信」で送信
5. **一斉送信完了後、参加者データの「送信日時」「送信ステータス」を必ず更新**
6. 参加者にQRコード付きHTMLメールが届く

### 4.3 フロントエンド（QR読み取りアプリ）機能

| 機能 | 内容 |
|------|------|
| QRコード読み取り | カメラでQRを読み取り、カーソルを合わせると受付成立 |
| モード切り替え | 画面上のボタン/スイッチで午前モード・午後モードを切り替え |
| 参加者情報表示 | GAS Web APIから取得したデータを表示 |
| 表示項目カスタマイズ | 午前・午後それぞれ表示項目を変更可能 |
| スキャン成功音 | 午前・午後どちらもスキャン成功時に再生 |
| **受付データ更新** | **スキャン時にGAS API経由で、現在のモード（午前/午後）に対応する受付ステータス・受付日時・受付担当者をスプレッドシートに必ず反映する** |
| **重複スキャン** | **同一参加者の複数回スキャンを許可する。現在のモードで既に受付済みの場合は「受付済み」と表示する（午前と午後は独立して判定する）** |
| **ID未登録時のエラー** | **QRコードのIDが参加者データに存在しない場合、エラー表示とエラー音を鳴らす** |

### 4.4 午前モード仕様

| 項目 | 内容 |
|------|------|
| 端末 | 職員のスマートフォン |
| スキャン主体 | 職員が学生のQRコードをスキャン |
| 表示項目（デフォルト） | ID、氏名、所属課程 |
| 用途 | 初回受付、本人確認、教室誘導 |

### 4.5 午後モード仕様

| 項目 | 内容 |
|------|------|
| 端末 | 受付窓口のパソコン |
| スキャン主体 | 学生本人がPCのウェブカメラにQRをかざす |
| 表示項目（デフォルト） | 氏名、集合教室 |
| 集合教室の値 | 参加者データシートの「集合教室」列をそのまま表示 |
| 用途 | 住民登録手続き時の本人確認、集合教室案内 |

### 4.6 共通仕様（フロントエンド）

| 項目 | 内容 |
|------|------|
| 表示項目のカスタマイズ | 午前・午後とも任意に変更可能 |
| スキャン成功音 | 両モードで再生 |
| エラー音 | IDが参加者データに存在しない場合に再生 |
| QRコード | 参加者IDをエンコードした同一QRコードを午前・午後で使用 |

## 5. GAS Web API アクセス制御（推奨方針）

**利用想定**: 当方の課内スタッフ約10名が利用。全スタッフが使えるようにする。

### 5.1 初心者向けの選択肢

GAS Webアプリのデプロイ時に「アプリにアクセスできるユーザー」を設定できます。

| オプション | 内容 | 推奨度 |
|------------|------|--------|
| **A. 全員** | リンクを知っている誰でもアクセス可能 | ★★★ 最も簡単 |
| **B. 組織内** | 同じGoogle Workspace組織のアカウントのみ | ★★ 組織利用なら |
| **C. 自分のみ** | デプロイした本人のみ | テスト用 |

### 5.2 推奨設定（課内10名スタッフ向け）

**オプションA「全員」** を推奨します。

- **理由**
  - 課内スタッフ全員が使える
  - 設定が簡単
  - 学内限定のURLであれば、リンクを知らない第三者にアクセスされるリスクは低い

- **運用上の注意**
  - WebアプリのURLはスタッフ間で共有し、外部に公開しない
  - URLが推測されにくい（GASが自動生成する長いURLのため、通常は問題になりにくい）

### 5.3 より厳格にしたい場合

- **オプションB「組織内」**: 大学のGoogle Workspaceアカウントを持つスタッフのみアクセス可能
- **パスワード追加**: アプリ起動時に簡単なパスワード入力を求める（実装が必要）

### 5.4 まとめ

まずは **「全員」** でデプロイし、URLを課内スタッフにのみ共有する運用で十分です。必要に応じて「組織内」やパスワード機能の追加を検討してください。

## 6. 非機能要件

| 項目 | 内容 |
|------|------|
| ホスティング | フロントエンドはGitHub Pages（無料・静的ホスティング） |
| オフライン | 参加者データ取得・受付登録はオンライン必須 |
| ブラウザ | カメラ利用のため HTTPS 必須（GitHub Pagesで対応） |
| 同時接続 | 窓口2口 × 午前・午後で想定される同時利用に対応 |

## 7. 運用フロー

### 7.1 事前準備

```
参加者データをスプレッドシートに入力
    ↓
メール設定（送信者・本文・QRコード等）をダイアログで設定
    ↓
「下書き一括作成」でGmailに下書きを作成
    ↓
下書き確認後「一斉送信」
    ↓
【一斉送信完了時】参加者データの「送信日時」「送信ステータス」を更新
    ↓
参加者にQRコード付きメールが届く
```

### 7.2 当日（午前）

```
9:30 受付開始
    ↓
職員がスマートフォンでQR読み取りアプリを起動（午前モード）
    ↓
学生がQRコードを提示
    ↓
職員がスキャン
    ↓
【IDが存在する場合】
  → ID・氏名・所属課程を表示
  → 午前受付ステータス・午前受付日時・午前受付担当者をスプレッドシートに更新
  → スキャン成功音
  → 午前モードで既に受付済みなら「受付済み」と表示（重複スキャン許可）
  → 職員が学生に氏名を確認 → 教室へ誘導
    ↓
【IDが存在しない場合】
  → エラー表示
  → エラー音
```

### 7.3 当日（午後）

```
住民登録手続き
    ↓
窓口のPCでQR読み取りアプリを起動（午後モード）
    ↓
学生本人がPCのウェブカメラにQRをかざす
    ↓
【IDが存在する場合】
  → 氏名・集合教室を表示
  → 午後受付ステータス・午後受付日時・午後受付担当者をスプレッドシートに更新
  → スキャン成功音
  → 午後モードで既に受付済みなら「受付済み」と表示（重複スキャン許可）
  → 本人確認 → 入室する教室を案内
    ↓
【IDが存在しない場合】
  → エラー表示
  → エラー音
```

## 8. 実装時の検討事項

| 項目 | 内容 |
|------|------|
| 受付担当者の識別 | スキャン時に「受付担当者」をスプレッドシートに記録するため、アプリ側で担当者を識別する必要がある。例：アプリ起動時に担当者名を選択するドロップダウン、または入力画面 |

## 9. エラー・例外処理

| 状況 | 動作 |
|------|------|
| QRコードのIDが参加者データに存在しない | エラー表示 + エラー音 |
| 既に受付済みの参加者をスキャン | 「受付済み」と表示 + スキャン成功音 + 受付データは更新（重複スキャン許可）。午前・午後は独立して判定する |
| ネットワークエラー | 適切なエラー表示（実装時に詳細を定義） |
| カメラアクセス拒否 | カメラ許可を促すメッセージ表示 |
